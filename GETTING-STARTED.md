# Getting Started

*See also: [README.md](README.md) · [docs/PHILOSOPHY.md](docs/PHILOSOPHY.md) · [docs/ADAPTING.md](docs/ADAPTING.md)*

Applying the harness to a new migration takes about 30 minutes of setup, most
of it filling in what your legacy app and target stack actually are.

## 1. Install the template into your repo

From your target repository root, run the installer:

```bash
bash /path/to/autonomous-work-harness/install.sh
```

On Windows, use the native PowerShell installer instead:

```powershell
& C:\path\to\autonomous-work-harness\install.ps1 -TargetDir .
```

If Windows PowerShell blocks unsigned local scripts, run it once with
`powershell -ExecutionPolicy Bypass -File C:\path\to\install.ps1 -TargetDir .`
or set an appropriate per-user execution policy.

It installs the whole `template/` tree at your repo root — `CLAUDE.md` (plus
the feature-profile variant `CLAUDE-feature.md`), `AGENTS.md`, `.claude/`,
`migration/`, `probes/`, `test/`, and `settings-permissions.patch` — makes the
scripts executable, and adds `.harness/` to your
`.gitignore` (local proof state, regenerated by `gates.sh` — not committed). It
refuses to overwrite existing files unless you pass `--force`, and prints a
status report at the end. Your legacy source stays where it is — you only point
the harness at it.

> **Repo already has a `CLAUDE.md` or `.claude/settings.json`?** `--force`
> overwrites wholesale — it does not merge. Back yours up first and fold your
> content back in by hand (the harness needs its hook wiring in
> `settings.json` and its hard rules in `CLAUDE.md`, but both coexist fine
> with your own additions).

Prefer to do it by hand? `cp -r /path/to/.../template/. .`, then
`chmod +x migration/tools/*.sh .claude/hooks/*.sh` and
`echo '.harness/' >> .gitignore`.

## 2. Configure `migration/harness.env`

This is the one file the hooks and scripts read. Settings:

- **`HARNESS_SCOPE`** — the paths whose changes require a gate proof. Include
  your new source tree, `migration`, `.claude`, and `CLAUDE.md` — **and the
  gates' own inputs**: build/CI scripts, CI config files, vendored tool
  binaries the gates execute, and the frozen oracle tree. A proof that does
  not cover the programs that define the proof can be laundered by editing
  those programs. Every entry must be stageable by git — an entry inside a
  gitignored directory makes the hash tool fail closed (it names the entry).
- **`HARNESS_FROZEN`** — path fragment(s) identifying the legacy oracle. Any
  edit under these is blocked. Point it at your legacy source (e.g.
  `legacy/src`, or `app/src model/src` for multiple). Leave it **empty** for
  an in-place migration — there is no separate legacy tree to freeze (step 3
  is then skipped).
- **`HARNESS_LOCKED`** — the harness's own enforcement files (gates, hooks,
  config). The default already covers them; you rarely change it. These are
  read-only to the agent so it can't weaken its own gates — a human edits them
  outside the agent session (which is also how you'd reconfigure `gates.sh`
  after the initial setup below).
- **`HARNESS_MAX_CALLS_PER_TICK`** — per-session tool-call budget (default
  200). When exceeded, a wrap-up warning is injected. Set to 0 to disable.
- **`HARNESS_LOOP_THRESHOLD`** / **`HARNESS_LOOP_WINDOW`** — loop detection:
  if the same tool+args fingerprint repeats `THRESHOLD` (default 3) times in
  the last `WINDOW` (default 6) calls, a reconsideration prompt is injected.
  Set threshold to 0 to disable.
- **`HARNESS_MAX_TICKS`** / **`HARNESS_LOCK_TTL_MIN`** — driver knobs for
  `kick-loop.sh`: the per-run tick budget for `--drive` (default 50) and the
  stale-lock recovery TTL in minutes (default 360). An environment variable on
  the invocation overrides the file value.

## 3. Record the frozen-oracle baseline (freeze-and-replace only, human, once)

`gates.sh` verifies on every run that the frozen oracle still hashes to a
committed baseline (`migration/frozen-baseline.sha`). Until that baseline
exists, the gate **fails closed** — an unverified oracle is unproven, not a
pass — so no gate run can succeed before this step. Recording it is a
one-time human step; the agent is deliberately blocked from `--record` (an
agent that can re-baseline can launder any drift it caused):

```bash
bash migration/tools/check-frozen.sh --record
git add migration/frozen-baseline.sha
git commit -m "record frozen-oracle baseline"
```

Doing an **in-place** migration (no separate legacy tree)? Leave
`HARNESS_FROZEN` empty in `harness.env` and skip this step —
`migration/tools/check-frozen.sh` then no-ops. (With `HARNESS_FROZEN` set to
a fragment that matches no files, it fails instead — a misconfiguration, not
a pass.)

## 4. Wire your real gates

Edit the **PROJECT GATES** block in `migration/tools/gates.sh`. It ships
deliberately failing so an unconfigured harness can't report a false pass.
Replace it with your stack's format-check + static analysis + FULL test suite
(examples for Dart, Node, Python, Go, Rust are in the file, and in
[docs/ADAPTING.md](docs/ADAPTING.md#gate-examples-by-stack)). The suite must
exit non-zero on any failure.

Then verify the plumbing:

```bash
bash migration/tools/gates.sh          # runs your gates; records proof once steps 3 and 6 are also done
cat .harness/state/gates-passed.diffsha # a hash appears on success
```

If `gates.sh` fails before it ever reaches your project gates, that is the
design, not a bug: it refuses while the frozen baseline is missing (step 3),
while `CLAUDE.md` still contains ship-time `<...>` markers (step 6), or while
the PROJECT GATES block is still the shipped stub.

## 5. Pre-authorize the agent's commands and edits

The shipped `.claude/settings.json` allow-lists the harness's own tools
(`gates.sh`, `kick-loop.sh`, `doctor.sh`, `check-docs.sh`,
`working-tree-hash.sh`, `check-complete.sh`, `check-frozen.sh`,
`record-audit.sh`, and the state tools). What it cannot ship is what only you
know: **your stack's build/test/format commands, and Edit/Write rules for the
tree the agent actually writes.** Add those BEFORE any unattended run — a
headless tick (`claude -p`, which is what `kick-loop.sh` runs) cannot show a
permission dialog, so a missing rule parks or fails the run instead of asking.

Two ways to add the Edit/Write (and fixture-generator) rules:

- **Apply the shipped patch.** Fill the `<TARGET_TREE>`,
  `<FIXTURE_GENERATOR>`, `<RUN_FIXTURE_GENERATOR>`, and `<BUILD_TEST_CMD>`
  placeholders in `settings-permissions.patch`, then from the repo root:
  `git apply settings-permissions.patch`. It adds Edit/Write allows for
  `migration/` and your target tree, a Bash allow for the fixture generator,
  and a Bash allow for your build/test tool (`<BUILD_TEST_CMD>` → e.g.
  `cargo`, `npm`, `go`).
- **Or hand-edit** `permissions.allow` in `.claude/settings.json` to the same
  effect.

Two easy-to-miss cases `doctor.sh` now flags for you:

- **The build/test tool must be allow-listed for DIRECT use.** The gate runs it
  via `bash migration/tools/gates.sh` (already allowed), but every tick also
  runs `cargo build`/`npm test`/… directly while iterating — each parks the
  loop if `Bash(cargo:*)` (etc.) isn't allowed. Add one `Bash(<tool>:*)` per
  tool the patch's single `<BUILD_TEST_CMD>` slot doesn't cover.
- **Root manifests live OUTSIDE your target tree.** `Edit(<TARGET_TREE>/**)`
  does not cover a repo-root `Cargo.toml` / `package.json` / `go.mod` /
  `pyproject.toml`, so creating or editing it parks the loop. Either add an
  explicit `Edit`/`Write` rule for the manifest, or put your target crate in a
  subdirectory so one glob covers it.

Do this as a human, or during the install session: `.claude/settings.json` is
`HARNESS_LOCKED`, so once the hooks are armed the agent cannot widen its own
permissions — see the bootstrap-window note below. `bash migration/tools/doctor.sh`
warns when a configured gate command isn't allow-listed, so run it before your
first unattended tick.

## 6. Fill in the placeholders

Search the template for `<...>` and replace. `gates.sh` refuses to pass while
`CLAUDE.md` still contains ship-time markers (e.g. `<PROJECT>`, `<LEGACY STACK>`),
so an unconfigured harness can't report a green slice — `doctor.sh` lists what
remains.

- `CLAUDE.md` — legacy/target stack names, repo layout, target architecture,
  and hard rules 5–6 (platform/perf constraints for your stack).
- `migration/PLAN.md` — phase breakdown for your codebase.
- `migration/legacy-runtime.md` — the verified build/run recipe for your
  legacy app and probes.
- `migration/SINGLE-TICK-PROMPT.md` and `migration/LOOP-PROMPT.md` —
  legacy/target names in the driver prompts.
- `settings-permissions.patch` — `<TARGET_TREE>`, `<FIXTURE_GENERATOR>`,
  `<RUN_FIXTURE_GENERATOR>`, `<BUILD_TEST_CMD>` (fill before `git apply`, step 5).

## 7. Optional: tighten the command guard

`.claude/hooks/pretooluse-command-guard.sh` blocks `--no-verify` universally.
Uncomment/adapt the project-specific guards (selective test filtering,
ungated dependency adds) for your test runner and package manager.

## 8. Run it

**Interactive, one slice at a time:**

```
/migrate-slice
```

**Unattended (recommended) — one fresh context per slice:**

```bash
bash migration/tools/kick-loop.sh --drive
```

Runs one slice per fresh headless session, back-to-back, until the migration
terminates (`migration/HANDOFF.md` written), a usage limit pauses it, or the
run's tick budget is spent. Headless sessions cannot answer permission
prompts — make sure step 5 is done first. Because no session accumulates context, quality
stays flat across a long migration. Fire it from a scheduler to survive
usage-limit resets — recipes in `migration/RESUMING.md`.

**Unattended, single self-paced session:** paste the contents of
`migration/LOOP-PROMPT.md` into a fresh Claude Code session at the repo root.
Use this only when you cannot run the driver above — a loop does not reset
context between iterations, so that prompt delegates each tick to a subagent to
get a fresh window per slice. It is the in-session approximation of `--drive`,
not an equal: the orchestrating session still has to live for the whole
migration, and it dies with the session. `--drive`'s driver is a shell script —
it has no context to fill and nothing to forget.

The first run should do Phase 0 (bootstrap): stand up `probes/`, generate the
first fixtures, scaffold the target project, and expand `parity-matrix.md` to
full coverage. Everything after that is one audited slice per pass.

## Check your setup any time

`doctor.sh` is a read-only report — it writes nothing, so run it whenever:

```bash
bash migration/tools/doctor.sh
```

```powershell
.\migration\tools\doctor.ps1
```

It shows your `HARNESS_*` config, whether `gates.sh` is still the ship-time stub,
any unfilled `<...>` placeholders, and — the useful one — whether the current
tree is **GATED** (covered by a recorded gate run), **STALE**, or has **NONE**.

## Sanity checks that the harness is live

> **Bootstrap window.** Hooks load at session start, so the Claude Code
> session that INSTALLS the harness is not yet governed by it — that session
> can still edit `gates.sh`, the hooks, and `harness.env`. This is the
> sanctioned moment to configure the locked files, whether a human or an
> agent is doing the setup; they arm for every following session. Restart
> after setup, then run these checks to confirm the hooks actually bite.

- Try to `Edit` a file under your frozen path → blocked.
- Change a file in scope, then end the turn without running gates → the Stop
  hook nags you to run `gates.sh`.
- Run `gates.sh` on a clean pass, then end the turn → allowed.

To verify the harness itself, run the full local test suite — syntax +
shellcheck + the runtime regression guard (content-addressed proof, the
Stop-hook challenge, the frozen/command/locked guards) + static consistency
checks + an end-to-end smoke test that installs the harness and drives a real
gate through pass/fail/re-gate. Needs bash ≥ 4, git, and GNU sed (see below):

```bash
bash test/run-all.sh
```

Run it after any edit to the hooks or `migration/tools/` scripts. `run-all.sh`
also runs in CI (`.github/workflows/`) on every push and PR. The harness's
*product* scripts (hooks, `migration/tools/`) run on stock macOS bash 3.2;
the *test suite itself* needs GNU sed and bash ≥ 4 (Linux and Git Bash have
both; on macOS `brew install bash gnu-sed` and run the tests with those). The suite keeps
working after you configure real gates: its throwaway repos replace the whole
PROJECT GATES block with test gates, so your project toolchain is never
invoked there.

### On Windows, run the TESTS under WSL — not Git Bash

The suite is fork-heavy by design (throwaway repos, a hook or a `git` call behind
almost every assertion), and Windows process creation is punishingly slow. Measured
on the same machine, same repo:

| | Git Bash (MinGW) | WSL2 |
|---|---|---|
| 200 process forks | 24 s | 0.16 s |
| one no-op `gates.sh` run | **44 s** | <1 s |
| the full self-test | >10 min | **30 s** |

Roughly 150× on forks. Nothing is wrong with your machine — that is just what
`fork()` costs on Windows.

```bash
wsl bash -c "cd /mnt/<drive>/path/to/repo && bash test/run-all.sh"
```

Your Windows drives are already mounted under `/mnt/`, so the repo does not need
to move (moving it onto WSL's own ext4 filesystem buys another ~8×, at the cost of
slow access from Windows tools).

Two Windows-only test failures are expected under Git Bash and are **not** real:
the symlink-aliasing guard cases. Creating a symlink on Windows needs privilege, so
MinGW's `ln -s` silently copies the file instead, leaving no symlink for the guard
to resolve. They pass under WSL and in CI (`ubuntu-latest`). If you see exactly
those two red, you are looking at the platform, not a defect.

**This applies to the tests only.** The unattended driver still requires Git Bash:
`kick-loop.sh` and `run-loop.ps1` need to see `claude.exe`, which WSL cannot — and
`run-loop.ps1` refuses a non-MINGW shell for exactly that reason. The test suite
has no such constraint because it drives a *fake* `claude` binary it builds itself.
